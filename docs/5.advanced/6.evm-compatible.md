# EVM networks

Most of the examples in this documentation are for Ethereum. But DipDup can index any EVM-compatible network as long as there's enough historical data. This page contains the list of supported EVM-compatible networks and a brief guide on how to configure them.

## Configuring datasources

A typical EVM indexer uses three types of datasources for different purposes. [evm.subsquid](../3.datasources/4.evm_subsquid.md) for historical data, [abi.etherscan](../3.datasources/1.abi_etherscan.md) for ABIs, and [evm.node](../3.datasources/3.evm_node.md) for the recent data and real-time subscriptions. Read the respective articles to learn more about them. If you've created a new project using `dipdup new` command and used one of EVM templates, `datasources` section in your config file should look like this:

```yaml [dipdup.yaml]
datasources:
  etherscan:
    kind: abi.etherscan
    url: ${ETHERSCAN_URL:-https://api.etherscan.io/api}
    api_key: ${ETHERSCAN_API_KEY:-''}

  evm_node:
    kind: evm.node
    url: ${NODE_URL:-https://eth-mainnet.g.alchemy.com/v2}/${NODE_API_KEY:-''}
    ws_url: ${NODE_WS_URL:-wss://eth-mainnet.g.alchemy.com/v2}/${NODE_API_KEY:-''}

  subsquid:
    kind: evm.subsquid
    url: ${SUBSQUID_URL:-https://v2.archive.subsquid.io/network/ethereum-mainnet}
    node: evm_node
```

Default values here are for Ethereum mainnet and Alchemy node provider. To configure datasources for other networks, you simply need to change URLs and API keys. You can do it in the config file directly, but it's better to use environment variables. Check the `deploy/.env.default` file in your project directory; it contains all the variables used in config.

```shell [deploy/.env.default]
ETHERSCAN_API_KEY=''
ETHERSCAN_URL=https://api.etherscan.io/api
NODE_API_KEY=''
NODE_URL=https://eth-mainnet.g.alchemy.com/v2
NODE_WS_URL=wss://eth-mainnet.g.alchemy.com/v2
SUBSQUID_URL=https://v2.archive.subsquid.io/network/ethereum-mainnet
```

### [abi.etherscan](../3.datasources/1.abi_etherscan.md)

Many explorers have Etherscan-like API and it could be used to retrieve ABIs for further types generating using `abi.etherscan` datasource.
You can get a URL for it in the table for your network, for some Etherscan-like explorers you will also need an API key.

```yaml [dipdup.yaml]
  etherscan:
    kind: abi.etherscan
    url: ${ETHERSCAN_URL:-https://api.etherscan.io/api}
    api_key: ${ETHERSCAN_API_KEY:-''}
```

When Etherscan-like API is not available, you can use ABI from the JSON file.

Put the JSON file with ABI to `abi/<typename>/abi.json` and run `dipdup init` to generate all necessary types, then you can run your indexer.

### [evm.subsquid](../3.datasources/4.evm_subsquid.md)

To configure `evm.subsquid` datasource, paste `evm.subsquid` URL from tables below or set `SUBSQUID_URL` variable in .env file.

```yaml [demo_evm_events/dipdup.yaml]
datasources:
  ...
  subsquid:
    kind: evm.subsquid
    url: ${SUBSQUID_URL:-https://v2.archive.subsquid.io/network/ethereum-mainnet}
    node: evm_node
  ...
```

```yaml [arbitrum/dipdup.yaml]
    ...
    url: https://v2.archive.subsquid.io/network/arbitrum-one
    ...
```

### [evm.node](../3.datasources/3.evm_node.md)

Node RPC API vary a lot across EVM networks, yet most of them supported by dipdup indexer to retrieve historical data.
Public node providers usually require API key, so demo_evm_events template allows you to pass HTTP URL, Websocket url, and API key into NODE_URL, NODE_WS_URL, and NODE_API_KEY using .env file to use node providers like Alchemy and Infura from the box.

```yaml [dipdup.yaml]
datasources:
  evm_node:
    kind: evm.node
    url: ${NODE_URL:-https://eth-mainnet.g.alchemy.com/v2}/${NODE_API_KEY:-''}
    ws_url: ${NODE_WEBSOCKET_URL:-wss://eth-mainnet.g.alchemy.com/v2}/${NODE_API_KEY:-''}
```
